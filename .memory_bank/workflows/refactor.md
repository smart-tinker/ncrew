# Workflow рефакторинга (Refactor)

## Цель
Улучшить структуру кода, удалить технический долг или мертвый код, сохраняя работоспособность системы.

## Принципы
- **TDD First:** Сначала тест (или фиксация текущего поведения), потом изменение.
- **Безопасность:** Изменения не должны ломать основной функционал (MVP).
- **Атомарность:** Один логический шаг за раз.

## Процесс

### 1. REQs & SPECs (Requirements & Specifications)
```bash
# Описать проблему (что не так с кодом/архитектурой).
# Создать файл спецификации в .memory_bank/specs/ с префиксом refactor_.
# Явно определить Scope: что удаляем, что меняем, что перемещаем.
```

### 2. Plan (Планирование)
```bash
# Разбить рефакторинг на безопасные шаги.
# Определить критерии успеха для каждого шага.
# Оценить влияние на существующие тесты.
```

### 3. Branching (Ветвление)
```bash
git checkout main
git pull origin main
git checkout -b refactor/description
```

### 4. Test Fail / Verify Baseline (Тестирование до изменений)
```bash
# ДЛЯ УДАЛЕНИЯ КОДА:
# 1. Запустить текущие тесты. Убедиться, что основной функционал (main.py) работает.
# 2. Убедиться, что тесты, покрывающие удаляемый код, существуют (если нет - это мертвый код без тестов).

# ДЛЯ ИЗМЕНЕНИЯ СТРУКТУРЫ:
# 1. Написать тест, который ожидает новую структуру (он должен упасть - RED).
# 2. Или зафиксировать текущее поведение снапшот-тестом.
```

### 5. Code (Реализация / Удаление)
```bash
# Выполнить изменение кода (удаление файлов, перемещение, переименование).
# Удалить или обновить тесты, которые были завязаны на удаляемый код.
```

### 6. Test Pass (Зеленый тест)
```bash
# Запустить полный тест-сюит.
# Все тесты должны пройти (GREEN).
# Убедиться, что покрытие кода (coverage) не упало критически в важных местах.
```

### 7. Review & Merge
```bash
# Self-review: не осталось ли "хвостов" (импортов удаленных файлов).
# Создать PR.
# Squash merge в main.
```
